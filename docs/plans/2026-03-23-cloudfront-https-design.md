# CloudFront HTTPS Design

## Problem

Current deployment outputs HTTP-only ALB URLs (`http://<alb-dns>`). All client-proxy traffic is unencrypted, which is a security concern especially since requests carry API keys and sensitive data.

## Solution

Add a CloudFront distribution in front of the existing ALB to provide HTTPS termination using AWS-managed `*.cloudfront.net` certificates. No custom domain required.

## Architecture

```
Client (Anthropic SDK)
    │
    ▼ HTTPS (443)
CloudFront (*.cloudfront.net)
    │  - AWS-managed TLS certificate
    │  - Attaches X-CloudFront-Secret header
    │
    ▼ HTTP (80, internal)
ALB (existing)
    │  - Validates X-CloudFront-Secret header
    │  - Rejects direct access with 403
    │
    ▼ HTTP (8000)
ECS Tasks (existing, unchanged)
```

## CloudFront Configuration

### Cache Behavior (Default)
- **Allowed Methods**: GET, HEAD, OPTIONS, PUT, POST, PATCH, DELETE
- **Cache Policy**: `CachingDisabled` (LLM responses are not cacheable)
- **Origin Request Policy**: `AllViewerExceptHostHeader` (forwards all headers including `x-api-key`, `anthropic-beta`)
- **Response Headers Policy**: Add HSTS header (`Strict-Transport-Security: max-age=31536000`)
- **Viewer Protocol Policy**: `redirect-to-https`

### Origin (ALB)
- **Protocol**: HTTP only
- **Origin Read Timeout**: 180 seconds (max, for extended thinking / slow first token)
- **Origin Keepalive Timeout**: 5 seconds (default)
- **Custom Header**: `X-CloudFront-Secret: <auto-generated-value>`

### Streaming (SSE) Compatibility
- CloudFront natively supports chunked transfer encoding
- SSE events are forwarded in real-time without buffering
- Origin Read Timeout (180s) only applies to time-to-first-byte; once streaming starts, data flows continuously

## ALB Security Hardening

When CloudFront is enabled, ALB listener rules are modified:
- **Match** `X-CloudFront-Secret` header → forward to target group (existing behavior)
- **No match** → return 403 Forbidden

This prevents direct access to the ALB, ensuring all traffic goes through CloudFront (HTTPS).

## Configuration

### New config options (`config/config.ts`)

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `enableCloudFront` | boolean | `true` | Enable CloudFront distribution with HTTPS |
| `cloudFrontSecret` | string | auto-generated | Secret header value for ALB validation |
| `originReadTimeout` | number | 180 | CloudFront origin read timeout in seconds |

### Behavior when `enableCloudFront: false`
- No CloudFront distribution created
- ALB accepts all traffic directly (current behavior)
- Deploy script outputs `http://` URLs

## Files Changed

| File | Change |
|------|--------|
| `cdk/config/config.ts` | Add CloudFront config options |
| `cdk/lib/ecs-stack.ts` | Add CloudFront Distribution + ALB header validation rule |
| `cdk/scripts/deploy.sh` | Output `https://` CloudFront URL |

## What's Excluded (YAGNI)

- WAF integration (Shield Standard provides baseline DDoS protection)
- Custom domain / ACM certificate
- Geographic restrictions
- CloudFront Functions / Lambda@Edge
- Multiple cache behaviors (admin portal and API share one behavior, both uncached)

## Rollback

Set `enableCloudFront: false` and redeploy. ALB header validation is removed, direct HTTP access is restored.
